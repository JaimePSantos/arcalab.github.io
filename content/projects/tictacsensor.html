Tic-Tac Sensor

(This project has been already selected)

This project will exploit a recent extension to Wireless Sensor Networks (WSNs) under development at KUL, which include a cheap but powerful distributed clock provided by a new network infrastructure. You will be expected to investigate how to map an application specified in a synchronous dataflow language into:

  • a set of deployment instructions, and
  • a set of rules for each node describing the local execution of the synchronous application.

Supervision

  • Promotor: Danny Hughes
  • Daily supervisor: José Proença

Context

Synchronous dataflow languages describe high-level and expressive communication patterns between concurrent components. However, these are typically computationally expensive to implement, especially in a distributed setting, making it a hard choice for specifying wireless sensor networks, where resources are scarce. Recent experiments with WSN use a network infrastructure that provide a global clock to all nodes in the network, usually either not available or complex to set-up. This global clock can be the key to define atomic actions over the network.

Currently we know how to define applications using LooCI component model and how to execute them over the LooCI middleware (which supports platforms such as Contiki and OSGI Java). LooCI can be extended with extra functionality, as done in other projects, to modify how events are exchanged between components.

Goal

The goal of this work is to support distributed and lightweight applications specified by rules that dictate when events are allowed to be sent, describing time and synchronisation constraints.

Possible research questions

  • What is the best way to represent synchronous data-flows in the context of sensor networks?
  • What are the different approaches to implement a synchronous system based on distributed components with a shared clock?
  • What kind of guarantees can we provide by exploiting the shared clock?
  • What are the costs (performance and/or energy) of imposing synchronisation constraints in such systems?

What the student will do

  • Development of a simple language for synchronous data-flows.
  • Encode such data-flows into deployable components.
  • Extend LooCI middleware to preserve the synchronous constraints.
  • Evaluate the cost of synchrony, for different design decisions.

Relevant literature

Student background and interests

  • Some familiarity with component models and automata.
